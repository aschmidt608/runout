- [ ] **Spotify support — built, then removed pending a Premium account**
  Built the full thing: OAuth Authorization Code + PKCE (same pattern as Tidal, no backend needed), connect/disconnect, silent token refresh (Spotify's PKCE flow does issue a `refresh_token`, unlike Tidal's), push-to-playlist (search by artist/title, create a private playlist, add tracks), and a merged "Connect a music service" tabbed panel with Spotify's official icon SVG.
  **Why it's pulled:** live testing hit "Your application is blocked from accessing the Web API since you do not have a Spotify Premium subscription." Confirmed this is a real, current Spotify policy, not a bug — as of their February 2026 Developer Mode changes, every Development Mode app now requires the *app owner's* account (not test users) to have an active Spotify Premium subscription, with no workaround for Free accounts; if the owner's Premium lapses, the app stops working entirely. That update also capped Development Mode at 5 users (down from 25) and 1 Client ID per developer, and renamed the add-tracks endpoint from `/playlists/{id}/tracks` to `/playlists/{id}/items` (fixed in code before the removal). Since the account this runs under isn't Premium, pulled the feature from `index.html` entirely (panel, OAuth functions, push flow, footer copy) rather than ship a permanently-broken "Connect Spotify" button — the Client ID (`ed434d68496647e19d59d0f34cf59268`) and this writeup are enough to rebuild it if that changes.
  Sources: [Update on Developer Access and Platform Security](https://developer.spotify.com/blog/2026-02-06-update-on-developer-access-and-platform-security), [February 2026 Web API Dev Mode Migration Guide](https://developer.spotify.com/documentation/web-api/tutorials/february-2026-migration-guide)
  Still need the official Tidal logo asset if the tabbed multi-DSP panel comes back — their developer portal's brand guidelines page is JS-rendered and couldn't be fetched programmatically, so that needs the account holder to log in and grab it directly rather than relying on unofficial third-party icon sites.

- [x] **YouTube Music support — researched, decided not to build**
  There's no official "YouTube Music" API — the only sanctioned option is the general YouTube Data API v3, which creates playlists of regular YouTube videos, not YouTube Music catalog tracks (matches would be a mix of official audio, music videos, live versions, and random reuploads, with no way to target clean audio-only tracks like Spotify/Tidal give us). The only thing that actually understands the YouTube Music catalog is `ytmusicapi`-style unofficial libraries, which replay browser cookies against Google's private internal endpoints instead of OAuth — not Google-sanctioned, a different risk profile than the other three integrations here.
  It also can't be a pure static-page PKCE flow like Tidal/Spotify: Google's "Desktop app" OAuth client type gives real no-secret PKCE but restricts redirect URIs to `http://127.0.0.1:port` (local-only, not usable on GitHub Pages); the "Web application" client type allows a real `https://` redirect URI but requires a `client_secret` at token exchange regardless of PKCE — meaning it'd need a small backend (same shape as the Discogs Worker) just to hold that secret.
  On top of that, quota is tight: search costs 100 units/call, playlist create and each item-add cost 50 units each, against a default 10,000 units/day — pushing one ~20-track set would cost roughly 3,050 units, so only ~3 pushes/day before hitting the ceiling without requesting a quota increase from Google.
  Given all that against Tidal + Spotify already covering the main use case well, decided to skip this rather than build a backend for a weaker, quota-constrained result.

- [x] **BPM enrichment for the Connect Discogs path — root cause found and fixed**
  GetSongBPM's API sends permissive CORS headers, so the lookup runs directly from the browser on `index.html` (no Worker needed there); the demo page proxies it through `claude-proxy-worker`'s `/demo/bpm` instead. The earlier "no result" / apparent outage was never a service problem or a bad key — it was `type=song`, which silently returns `{"search":{"error":"no result"}}` for *any* query regardless of formatting. `type=both` is the correct value and returns real matches in the exact shape the code already expected (`search[0].tempo`) — confirmed live against both a mainstream track (Billie Jean, tempo 116) and an obscure one (Voodoo Ray, tempo 123, matches its actual BPM). Fixed in `index.html`, `fetch_discogs_collection.py`, and `claude-proxy-worker/src/worker.js` (needs a manual redeploy via the Cloudflare dashboard to take effect there, same as any other Worker code change).
  **Follow-up: made opt-in instead of automatic.** Real usage on a large collection showed well under 30% match rate, and the lookup takes ~20 minutes — forcing that wait on every first sync was the wrong default. "Refresh collection" and "Load manually" are fast again (they still carry forward BPM values already found in a previous sync, so nothing already matched is lost) but no longer run new lookups automatically. A separate "Look up BPM" button, with an explicit disclaimer about coverage and slowness, runs enrichment on-demand against whatever's currently loaded, reusing the same incremental "only look up releases missing a bpm" logic.
  **Follow-up: mobile backgrounding.** Added the Screen Wake Lock API (`navigator.wakeLock.request('screen')`, well-supported now — iOS Safari 16.4+, Chrome 84+) during `enrichBpm()`, released when it finishes, re-acquired on `visibilitychange` per MDN's recommended pattern. This genuinely fixes the "phone auto-locks the screen" case. It does *not* fix switching away to another app — that's the browser/OS throttling or fully suspending background tab JS to save battery, a deliberate platform policy with no client-side override. Disclaimer text now says this explicitly rather than implying the whole problem is solved.
  **Follow-up: added a free, keyless second source.** Confirmed Spotify's old Audio Features API (tempo/key with zero download, the ideal "fully online" option) is dead — deprecated November 27, 2024, 403 for any app created since. But AcousticBrainz's *read* API is still live (confirmed by direct request) despite the project freezing new submissions in 2022 — it's a static archive keyed by MusicBrainz ID, not text search, so `enrichBpm()` now falls back to MusicBrainz's free recording search (paced at 1.1s/call, its documented ~1 req/sec per-IP limit; only accepts matches scoring ≥90 to avoid assigning the wrong track's BPM) to resolve an MBID, then queries AcousticBrainz's `rhythm.bpm` for that MBID. Both are CORS-open, so this runs entirely client-side with no Worker changes. Since this path needs no key at all, `index.html`'s GetSongBPM key is now optional rather than required — the free fallback always runs, GetSongBPM runs first in addition if a key is set. Status message reports which source found each match. Live spot-check on two mainstream tracks (Blue Monday, Voodoo Ray) got confident MusicBrainz matches but 404s from AcousticBrainz for those specific recordings — confirms the mechanism works, but also confirms the archive really is sparse; actual combined coverage improvement is unverified until run against a real collection.

- [ ] **Idea: replace/supplement GetSongBPM with real audio analysis** — not started, needs more thought
  Given GetSongBPM's coverage is often under 50%, the idea is to instead pull an actual audio (or video) file for each release and analyze tempo (and maybe musical key) directly, rather than depending on a third-party lookup database. Floated approach: a tool like `streamrip` (uses authenticated DSP sessions to download audio) to grab a lower-quality file, then run it through an audio-analysis library to extract tempo/key.
  Worth noting before building anything: **`whisper` isn't the right tool here** — it's speech-to-text, not tempo/key detection. The correct category is a music information retrieval library, e.g. `librosa` (Python, well-established, does both tempo estimation and key/chroma detection) or `essentia`/`aubio` as alternatives.
  **Real tradeoffs to weigh:**
  - Downloading audio via unofficial tools against a DSP's authenticated session (streamrip-style, or yt-dlp for YouTube) is against those services' Terms of Service — a real consideration even for personal-use analysis of tracks you already have legitimate access to, and a bigger one if this were ever built into the shared demo page rather than kept as a personal local tool.
  - YouTube's catalog is broader for rare/vinyl-only material where Tidal's licensing has gaps, per your point — worth confirming in practice how well YouTube search actually finds obscure vinyl rips before relying on it.
  - Architecturally this is a different shape of project than Runout's current one: audio download + decode + FFT-based analysis is compute-heavy and needs a real runtime (Python + librosa), which Cloudflare Workers can't run — this would need its own local script or a proper server/backend, not a fit for the current static-page-plus-stateless-Worker design.
  Given all that, this is plausibly its own separate tool/project (a local script producing a BPM/key dataset Runout could import, similar in spirit to `fetch_discogs_collection.py`) rather than something to build directly into Runout's existing architecture.
  **Real-world precedent found:** [Loooom's "identify song parts" skill](https://www.mager.co/blog/2026-03-26-loooom-identify-song-parts/) confirms the analysis approach — it uses `ffmpeg` + `librosa` (not `whisper`, matching the note above) to extract tempo, key, and even song structure (intro/verse/chorus/outro with timestamps) and stem separation, all by having Claude Code write and run the analysis script against a local audio file. It confirms the analysis side is very achievable, but it operates on an MP3 you already have — it doesn't touch the acquisition problem (downloading audio for tracks in the collection), which is still the part with real ToS exposure. Cheapest next step if this gets picked back up: manually test the same ffmpeg+librosa approach against a couple of known tracks to validate real accuracy before building anything.

- [x] **DSP account refresh — resolved, not buildable as originally scoped**

- [x] **Fix Discogs 429 rate-limit bug + harden JSON parsing on the OAuth worker calls**
  Done. `fetchDiscogsCollection()` had no re-entrancy guard, so double-clicking "Refresh collection" (or a slow first page) could fire overlapping request batches and trip Discogs' rate limit — fixed with a `discogsFetchInFlight` flag that also disables the Refresh button mid-fetch. Separately hardened all three Discogs Worker calls (`connectDiscogs`, `handleDiscogsCallback`, `discogsApiCall`) to use a `safeJson()` helper that gives a clear error instead of a cryptic "Unexpected end of JSON input" if a response body is ever empty/non-JSON.
  **Important discovery:** Discogs' `/oauth/request_token` rate limit appears to be scoped to the app's Consumer Key (or the Worker's shared egress IP), not per-user — so it can get tripped by cumulative testing traffic across everyone using this app, and stays tripped as long as *anyone* keeps retrying through it.
  **Follow-up, now shipped:** this became a real problem for an actual large collection (2032 releases, ~21 sequential Discogs pages) — a 429 partway through used to abort the whole fetch and lose all progress. `discogsApiCall` now attaches the HTTP status to thrown errors, and a new `discogsApiCallWithRetry` wraps the pagination loop: on 429 it waits (8s/16s/24s) and retries the same page up to 3 times, with a visible "Rate limited by Discogs — waiting Ns" status, before giving up. Also widened page-to-page pacing from 1.1s to 1.5s for more headroom.

- [x] **Gated "try it with my API key" demo version for friends** — deployed and live-verified
  Done as designed: a real client-side password check alone doesn't protect an embedded API key (anyone can view-source a static page and read it out), so this needed a server-side proxy rather than just shipping the raw key. Built `claude-proxy-worker/` (deployed via the Cloudflare dashboard, not wrangler — no CLI installed) which holds the Anthropic API key as a secret, checks a shared demo password per request (with a `/demo/verify` endpoint so the page can confirm the password immediately instead of only failing deep in a generate attempt), forwards to Anthropic's Messages API, and enforces a daily generation cap (default 50) via Workers KV so a friend can't run up an unbounded bill. Also proxies GetSongBPM lookups via `/demo/bpm` the same way, so demo users don't need their own GetSongBPM key either (lower-stakes key, but same reasoning). `demo.html` is a variant of `index.html` with the "Claude API key" panel replaced by a "Demo access" password panel and no GetSongBPM key field — everything else (Discogs, Tidal, filters, saved sets) is identical. Confirmed live end-to-end on GitHub Pages. Not linked from the main page — reachable only via direct URL, same as any other GitHub Pages file.
  **Resolved:** GitHub Pages' legacy build system failed to auto-trigger a rebuild after a push three times this session (page kept 404ing / serving stale content, build stuck at `status: building, duration: 0`), each fixed with a manual `gh api --method POST .../pages/builds`. Switched the repo to GitHub Actions-based Pages deployment (`.github/workflows/pages.yml` + `build_type: workflow`) instead — every push now runs a real, visible Actions workflow, no more manual kicks needed.
