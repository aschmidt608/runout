- [ ] **Spotify support** — code done, Client ID set, blocked on account requirement
  Same OAuth Authorization Code + PKCE pattern already proven out for Tidal (no backend needed). Connect/disconnect, push-to-playlist (search by artist/title, create a private playlist, add tracks), and a merged "Connect a music service" tabbed panel (replacing the separate always-shown Tidal/Spotify panels) are all built. Confirmed this is a real, current Spotify policy, not a bug — as of their February 2026 Developer Mode changes, every Development Mode app now requires the *app owner's* account (not test users) to have an active Spotify Premium subscription, with no workaround for Free accounts; if the owner's Premium lapses, the app stops working entirely. Same update also capped Development Mode at 5 users (down from 25) and 1 Client ID per developer, and renamed the add-tracks-to-playlist endpoint from `/playlists/{id}/tracks` to `/playlists/{id}/items` (our code's been updated to the new path). Nothing left to build here until the account itself has Premium — this is a hard account-level gate, not something to code around.
  Sources: [Update on Developer Access and Platform Security](https://developer.spotify.com/blog/2026-02-06-update-on-developer-access-and-platform-security), [February 2026 Web API Dev Mode Migration Guide](https://developer.spotify.com/documentation/web-api/tutorials/february-2026-migration-guide)
  Tab icons: using Spotify's official icon SVG (downloaded from developer.spotify.com's own brand-assets zip). Still need the official Tidal logo asset — their developer portal's brand guidelines page is JS-rendered and couldn't be fetched programmatically, so this needs the account holder to log in and grab it directly rather than relying on unofficial third-party icon sites.

- [x] **YouTube Music support — researched, decided not to build**
  There's no official "YouTube Music" API — the only sanctioned option is the general YouTube Data API v3, which creates playlists of regular YouTube videos, not YouTube Music catalog tracks (matches would be a mix of official audio, music videos, live versions, and random reuploads, with no way to target clean audio-only tracks like Spotify/Tidal give us). The only thing that actually understands the YouTube Music catalog is `ytmusicapi`-style unofficial libraries, which replay browser cookies against Google's private internal endpoints instead of OAuth — not Google-sanctioned, a different risk profile than the other three integrations here.
  It also can't be a pure static-page PKCE flow like Tidal/Spotify: Google's "Desktop app" OAuth client type gives real no-secret PKCE but restricts redirect URIs to `http://127.0.0.1:port` (local-only, not usable on GitHub Pages); the "Web application" client type allows a real `https://` redirect URI but requires a `client_secret` at token exchange regardless of PKCE — meaning it'd need a small backend (same shape as the Discogs Worker) just to hold that secret.
  On top of that, quota is tight: search costs 100 units/call, playlist create and each item-add cost 50 units each, against a default 10,000 units/day — pushing one ~20-track set would cost roughly 3,050 units, so only ~3 pushes/day before hitting the ceiling without requesting a quota increase from Google.
  Given all that against Tidal + Spotify already covering the main use case well, decided to skip this rather than build a backend for a weaker, quota-constrained result.

- [x] **DSP account refresh — resolved, not buildable as originally scoped**
  Confirmed: Discogs' OAuth 1.0a tokens never expire (valid until manually revoked), so there's nothing to refresh there — the footer copy was just wrong and has been corrected. Tidal's token exchange genuinely returns no `refresh_token` (confirmed empirically) for a real registered third-party PKCE app, unlike unofficial libraries (e.g. `tidalapi`) that appear to borrow different/native client credentials — so there's no refresh token to silently renew with, and reconnecting after Tidal's ~4hr expiry requires the full interactive login same as a manual reconnect. If this becomes annoying in practice, the next idea would be a smoother one-click reconnect prompt rather than actual silent refresh, since silent refresh isn't available.

- [x] **Filter by media format (7"/45s vs. 12"/LPs, etc.)**
  Done. Confirmed empirically (via real Discogs release payloads) that `formats[].descriptions` already carries exactly this data (e.g. `["7\"", "45 RPM", "Single"]` vs `["12\"", "33 ⅓ RPM"]`) — both `fetch_discogs_collection.py` and the Connect Discogs OAuth path now capture it as `formatDescriptors`. - [x] **Fix Discogs 429 rate-limit bug + harden JSON parsing on the OAuth worker calls**
  Done. `fetchDiscogsCollection()` had no re-entrancy guard, so double-clicking "Refresh collection" (or a slow first page) could fire overlapping request batches and trip Discogs' rate limit — fixed with a `discogsFetchInFlight` flag that also disables the Refresh button mid-fetch. Separately hardened all three Discogs Worker calls (`connectDiscogs`, `handleDiscogsCallback`, `discogsApiCall`) to use a `safeJson()` helper that gives a clear error instead of a cryptic "Unexpected end of JSON input" if a response body is ever empty/non-JSON.
  **Important discovery:** Discogs' `/oauth/request_token` rate limit appears to be scoped to the app's Consumer Key (or the Worker's shared egress IP), not per-user — so it can get tripped by cumulative testing traffic across everyone using this app, and stays tripped as long as *anyone* keeps retrying through it. If this becomes a recurring problem for real users (not just during dev testing), worth adding a visible cooldown/backoff message rather than a generic connection-failed error.
